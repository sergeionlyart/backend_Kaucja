

Workflow: Main (Product) + Labs (Experiments) через Git Worktree

Преамбула

Цели
	1.	Чистый продукт: основная ветка и рабочая директория содержат только то, что относится к микросервису и его поддержке.
	2.	Изолированные эксперименты: прототипы/PoC/исследования ведутся отдельно и не “засоряют” продуктовую историю.
	3.	Воспроизводимость: полезные инструменты и артефакты разработки сохраняются так, чтобы их можно было запускать на любой машине.
	4.	Прозрачность и масштабируемость: понятные правила “куда класть” и “как переносить” результат, без хаоса из папок-проектов.

Принципы
	•	Разделение по назначению: продукт ≠ инструменты ≠ эксперименты ≠ локальные черновики.
	•	Main всегда зелёный: main собирается/тестируется и готов к релизу.
	•	Эксперименты не мерджатся в main: перенос только “выжимкой” (cherry-pick/переписывание), без мусора.
	•	Физическое разделение рабочих деревьев: продукт и лаборатория живут в разных директориях через git worktree.

⸻

1) Термины и зоны ответственности
	•	Product (микросервис) — код, тесты, конфиги, необходимые для работы и поставки сервиса.
	•	Tools (dev-инструменты) — утилиты, которые полезны для разработки/поддержки продукта и должны быть воспроизводимы.
	•	Labs (эксперименты) — PoC, исследования, бенчмарки гипотез, альтернативные реализации, “пощупать идею”.
	•	Scratch (локальные черновики) — одноразовые скрипты и артефакты “на сейчас”, не предназначены для репозитория.

⸻

2) Структура репозитория в main

Рекомендуемый каркас:

repo/
  service/            # продукт (микросервис)
  tools/              # воспроизводимые dev-инструменты (коммитим)
  docs/               # документация (опционально)
  infra/              # инфраструктура (опционально)

  scratch/            # локальное (НЕ коммитим)

Правила:
	•	service/ — единственный источник поставляемого кода.
	•	tools/ — коммитим только то, что:
	•	используется > 1 раза, или
	•	нужно команде, или
	•	важно для поддержки продукта.
	•	scratch/ — всегда в .gitignore, никогда не пушим.

⸻

3) Ветки: политика и ограничения

Обязательные ветки
	•	main — продуктовая ветка (protected).
	•	labs — ветка для экспериментов (orphan), без мерджа в main.

Ветки разработки
	•	feature/<ticket>-<slug> — фичи/фиксы продукта (от main).
	•	hotfix/<ticket>-<slug> — срочные фиксы (от main).
	•	(опционально) exp/<slug> — временные ветки экспериментов (от labs), удаляются после завершения.

Жёсткие правила
	•	В main нельзя коммитить:
	•	прототипы/PoC, ноутбуки, “грязные” тесты гипотез,
	•	разовые скрипты без поддержки,
	•	большие артефакты данных/выгрузок.
	•	labs не мерджится в main (никаких PR labs -> main).

⸻

4) “Две директории рядом” через git worktree

Однократная настройка labs как orphan-ветки

В корне репозитория:

git checkout --orphan labs
git rm -rf .
echo "# Labs" > README.md
git add README.md
git commit -m "Init labs branch"
git push -u origin labs
git checkout main

Создание соседней директории для лаборатории

На уровне, где лежит repo/:

cd repo
git worktree add ../repo-labs labs

Итог:
	•	repo/ — рабочее дерево main (продукт).
	•	repo-labs/ — рабочее дерево labs (эксперименты).

⸻

5) Правила ведения labs

Структура экспериментов

Каждый эксперимент — отдельная папка с README:

repo-labs/
  YYYY-MM-DD__topic__goal/
    README.md
    src/...
    notes.md (опционально)

Минимум в README.md эксперимента (5 пунктов)
	1.	Цель и гипотеза (1–3 строки).
	2.	Как запустить (команды).
	3.	Критерий успеха (что считаем “взлетело”).
	4.	Результат (факты/цифры/артефакты).
	5.	Решение: “в продукт / повторить позже / закрыть”.

Допуск

В labs можно:
	•	быстро менять структуру,
	•	держать PoC зависимости отдельно,
	•	хранить черновые идеи.

Но всё равно:
	•	никакого “мусора без контекста”: у каждого эксперимента должен быть README.

⸻

6) Инструменты разработки (tools/) — требования

Инструмент можно положить в tools/, если:
	•	он воспроизводим (понятно как запускать),
	•	он не ломает сборку продукта,
	•	он не попадает в релиз.

Рекомендация по зависимостям:
	•	либо общий инструментальный env (если это принято),
	•	либо каждый инструмент со своим окружением (часто лучше для изоляции).

⸻

7) Защита от “утечки” в релиз

.gitignore (минимум)

/scratch/
/data/
/.artifacts/
*.ipynb_checkpoints

.dockerignore (минимум)

tools/
scratch/
docs/
infra/
data/
.artifacts/
.git/

Docker/CI правило

Сборка/упаковка должна брать только service/. Любые “вспомогательные” директории не должны влиять на прод-артефакты.

⸻

8) Перенос результата из labs в main

Запрещено:
	•	merge labs в main.

Разрешено (рекомендуется):
	1.	Cherry-pick только чистых коммитов, которые относятся к продукту.
	2.	Перенос “вручную” (копированием) с последующим нормальным рефакторингом в service/.
	3.	В PR обязательно:
	•	объяснить источник (“из эксперимента YYYY-MM-DD__…”),
	•	оставить ссылку/путь на эксперимент в labs (для контекста).

Правило качества:
	•	перед переносом результат приводится к продуктовым стандартам (архитектура, тесты, линтеры, конфиги).

⸻

9) Правила коммитов и PR для main
	•	В main — только через PR.
	•	PR должен:
	•	проходить тесты/линтеры,
	•	иметь короткое описание “что/зачем/как проверить”,
	•	не тащить нерелевантные изменения (форматирование отдельно от логики по возможности).
	•	main должен быть “зелёным” всегда.

⸻

10) Быстрый чеклист “куда класть”

Это одноразовый черновик? → scratch/ (не коммитим)
Это инструмент, который пригодится ещё? → tools/ в main
Это PoC/исследование/гипотеза? → labs (через repo-labs/)
Это часть продукта? → service/ + PR в main
